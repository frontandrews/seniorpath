## The problem

Many tasks look simple until you notice nobody aligned what actually needs to come out of the work.

The ticket exists, but the goal, boundary, and definition of done are still unclear.

When that slips through, the team can stay busy and still remain misaligned.

## Mental model

A ticket is not only a list of things to do.

It is an execution decision.

The useful question is:

> What should this work produce, what can go wrong, and what is the real next step?

That pulls the task toward clarity instead of volume.

## Breaking it down

A simple way to think about a ticket is:

1. state the goal in one sentence
2. separate essential scope from optional scope
3. name the main risk or uncertainty
4. turn it into one visible next step

That prevents work that looks active but stays vague.

## Simple example

Imagine a ticket that only says:

> Improve onboarding.

Almost anything fits inside that.

A more useful version would say:

- goal: reduce drop-off in the first step
- current scope: review the initial form and its error message
- out of scope: redesign the entire flow
- risk: dependency on the validation API

Now the work becomes execution instead of a theme.

## Common mistakes

- accepting a ticket that is too broad without shaping it
- mixing problem, solution, and future improvement in one block
- starting implementation before agreeing on what done means
- treating backlog text as naturally clear by default

## How a senior thinks

A senior engineer does not receive a task passively.

They shape it before spending energy:

> Before I start executing, I want the goal, the boundary, and the main risk to be explicit. That prevents movement without direction.

## What the interviewer wants to see

Interviewers usually want to know:

- you can turn a loose request into an executable plan
- you can see risk before implementation starts
- you can separate present scope from future desire

That signals lower noise and less rework.

> A weak ticket does not improve on its own. Someone has to give it shape.

> If the next step is still blurry, the work is probably not ready to start.
